# 위험의 기원: 템플릿의 구조적 문제인가, 에이전트의 근본적 한계인가?

사용자께서 혼란스러워하시는 부분인 "제기된 위험들이 **에이전트라는 개념 자체**의 문제인지, 아니면 **현재 구현된 `claude-vibe-flow`**의 문제인지"를 명확히 구분하여 답변 드립니다.

결론부터 말씀드리면, **"AI 에이전트의 일반적인 한계가 현재 템플릿의 '다중 에이전트 구조'와 만나 증폭된 상태"**라고 볼 수 있습니다.

---

## 1. 현재 템플릿(`claude-vibe-flow`)의 구조적 특징에 기인한 점
이 부분은 템플릿의 설계를 변경함으로써 충분히 개선 가능한 영역입니다.

*   **유지보수의 역설 (심각도: 상)**: 
    - **원인**: 15개라는 많은 수의 에이전트 분절화.
    - **설명**: 에이전트가 단일하거나 3~4개였다면 발생하지 않았을 '에이전트 간 설정 동기화' 및 '관리 피로도' 문제입니다. 이는 현재 템플릿의 **'고도로 분업화된 설계'**에서 오는 부작용입니다.
*   **창의적 직관의 차단 (심각도: 상)**:
    - **원인**: `planner.md` 등에 심어놓은 '소크라테스식 질문' 프롬프트.
    - **설명**: 에이전트가 질문을 많이 하도록 **사용자가 직접 가이드**했기 때문에 발생하는 현상입니다. 프롬프트를 "질문 대신 바로 실행" 위주로 고치면 해결 가능합니다.
*   **오버엔지니어링 (심각도: 중)**:
    - **원인**: `architect.md`와 `spec-validator.md`의 엄격한 프롬프트.
    - **설명**: 에이전트에게 "정석적인 설계"를 강요하고 있기 때문에 발생합니다. 토이 프로젝트용으로 프롬프트를 경량화하면 개선됩니다.

---

## 2. AI 에이전트/LLM의 근본적인 한계(에이전트 자체의 문제)
이 부분은 어떤 템플릿을 쓰더라도, 심지어 직접 코딩을 도와달라고 해도 발생할 수 있는 AI의 숙명적인 문제입니다.

*   **가짜 안전망 (심각도: 중)**:
    - **설명**: "AI가 짠 코드를 AI가 검증할 때 발생하는 논리적 편향"은 에이전트가 1개든 100개든 발생합니다. 이는 현재 LLM 기술이 가진 **'자기 확신적 편향(Self-consistency bias)'**의 문제입니다.
*   **비즈니스 맥락 이해 부족**:
    - **설명**: 코드가 기술적으로(타입, 린트) 완벽해도 실제 비즈니스 로직(예: 할인율 계산이 상식적인가?)을 놓치는 것은 모든 AI 에이전트의 공통된 한계입니다.

---

## 3. 요약 및 해결 방향

| 위험 요소 | 기원 (Origin) | 해결 가능성 |
| :--- | :--- | :--- |
| **관리 피로도** | **템플릿 구조 (15개 분업)** | 에이전트 통합(Merge)으로 해결 가능 |
| **질문 피로도** | **템플릿 프롬프트 (Socratic)** | 프롬프트 스타일 수정으로 해결 가능 |
| **오버엔지니어링** | **템플릿 가이드 (정석 강조)** | 경량 설계 가이드로 수정 가능 |
| **검증 편향성** | **에이전트 근본 한계** | 외부 도구(TestRunner 등) 연동으로 보완 |
| **토큰 비용** | **구조 + 근본 한계** | 에이전트 축소 및 효율적 프롬프트로 완화 |

### 그래서 어떻게 하면 좋을까요?
사용자의 `claude-vibe-flow`는 **매우 훌륭한 "전문가 집단" 모델**입니다. 다만, 토이 프로젝트를 위해서는 이 전문가 집단이 너무 '딱딱하고 관료적'인 것이 문제입니다.

**개선 방향**:
1.  **에이전트 통폐합**: 15개를 3~5개의 '멀티 태스킹 에이전트'로 합치세요. (예: `planner`+`architect` → `concept-master`)
2.  **프롬프트 톤앤매너 변경**: "질문 우선"에서 **"실행 우선, 모호할 때만 질문"**으로 프롬프트의 철학을 바꾸면 사이드 프로젝트에 훨씬 최적화된 도구가 될 것입니다.
